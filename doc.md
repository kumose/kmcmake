# 约束即正确：kmcmake 与 AI 时代的构建规则系统

## 摘要

**构建系统的错误，绝大多数来源于表达空间过大导致的非确定性选择问题。** kmcmake 不是“简化 CMake”，也不是一套“设计哲学”——它是一个**可执行的规则系统**。它将管理意志直接编码为工具约束，将“应当怎样写”的文档规范，转化为“只能这样写”的强制规则。kmcmake 剥夺了 CMake 的图灵完备性，将构建脚本从“可编程语言”降级为**刻意不完备的声明式子语言**。在这个子语言中，没有结构级控制流、没有动态目标生成、没有运行时文件探测。所有构建意图必须通过有限种类的声明宏表达，所有隐性条件必须显性化为宏参数。kmcmake 的价值不是“提供便利”，而是**工具即制度**——通过工具本身的约束力，消灭非确定性，让正确成为唯一可构建的路径。无论人类开发者还是 AI 助手，都无需阅读冗长的规范文档，因为“规范”已经被硬编码在工具里。本文揭示 kmcmake 作为规则系统的本质，并展示这一范式如何统一配置管理（xconfig）、构建系统（kmcmake）、SQL 优化（Goose）等不同领域。

---

## 一、起点：文档规范 vs 工具约束

几乎所有团队都有构建系统的“规范文档”：规定必须使用 `add_library` 而非 `file(GLOB)`，规定必须设置 `POSITION_INDEPENDENT_CODE`，规定依赖链接的作用域……但这些文档很快就会被遗忘、被违反、被绕过。因为**文档没有执行力**。

kmcmake 的出发点是：**把规范写进工具里，而不是写进文档里**。它不再依赖开发者的自觉，而是通过工具本身的结构强制规范。你无法在 kmcmake 中写 `file(GLOB)`，因为根本就没有这个命令。你无法在 kmcmake 中写结构级 `if`，因为宏的参数系统不提供这种能力。你无法动态生成 target 名称，因为 target 必须在宏调用时静态确定。

这就是 **工具即制度**——制度不再是纸面上的文字，而是嵌入工具中的不可绕过约束。

## 二、CMake 的图灵完备性诅咒

CMake 是一个强大的构建系统生成器，但它本质上是**一门图灵完备的脚本语言**。这意味着，你可以用它写循环、分支、递归、甚至无限循环。这种表达能力带来的不是自由，而是混乱。

同一个“构建一个库”的意图，在 CMake 中有几十种等价写法：

```cmake
# 方式1：直接 add_library
add_library(mylib mylib.cc)
target_link_libraries(mylib PUBLIC Threads::Threads)
target_include_directories(mylib PUBLIC include)

# 方式2：先设置变量再 add
set(SOURCES mylib.cc)
set(LIBS Threads::Threads)
add_library(mylib ${SOURCES})
target_link_libraries(mylib PUBLIC ${LIBS})

# 方式3：使用 GLOB
file(GLOB_RECURSE SRC "src/*.cc")
add_library(mylib ${SRC})
```

这种图灵完备性带来的问题：
- **非确定性**：同一个意图对应无数条路径，构建行为难以预测。
- **隐性条件泛滥**：编译选项、平台适配、工具链差异隐藏在条件分支中。
- **对 AI 不友好**：AI 需要在指数级的可能性中猜测正确写法，极易产生幻觉。
- **维护灾难**：大型项目的 CMake 脚本动辄上万行，成为另一种“编程语言”。

kmcmake 的应对不是“简化”，而是**剥夺**——剥夺 CMake 的图灵完备性，将其降级为**刻意不完备的声明式子语言**。

## 三、规则系统的三个层级

kmcmake 构建了一个完整的规则系统，从最外层约束到最内层决策，层层递进：

### 层级 1：语言层 —— 剥夺图灵完备性

传统 CMake 是一门图灵完备的脚本语言，表达空间无限。kmcmake 将其降级为**刻意不完备的声明式子语言**：

| 能力 | 原生 CMake | kmcmake |
|------|-----------|---------|
| 结构级控制流 | ✅ if/foreach | ❌ 禁止 |
| 动态 target 生成 | ✅ 支持 | ❌ 禁止 |
| 运行时文件探测 | ✅ GLOB | ❌ 禁止 |
| 变量拼接逻辑 | ✅ 任意 | ❌ 禁止 |

这不是“简化”，而是**主动放弃能力**。这种放弃是故意的、激进的、不可协商的。它从语言层面消除了整类错误。

### 层级 2：宏层 —— 封闭表达空间

在有限的表达空间内，kmcmake 提供了有限种类的声明宏：
- `kmcmake_cc_library`：唯一方式声明库
- `kmcmake_cc_binary`：唯一方式声明可执行文件
- `kmcmake_cc_test`：唯一方式声明测试
- `kmcmake_cc_proto_object`：唯一方式处理 Protobuf

每个宏的参数是**结构化的、类型化的**，不存在多种等价写法。表达空间被压缩到 O(1)。

### 层级 3：规则层 —— 管理意志下沉

这是 kmcmake 最深刻的设计：**将管理决策从文档下沉到工具**。传统团队需要通过代码审查、文档培训来确保构建脚本规范，而 kmcmake 让这些规范变得**不可违反**。

例如，传统规范文档会写：
> “推荐使用显式源文件列表，避免使用 GLOB。”
> “推荐使用 PUBLIC/PRIVATE 明确链接作用域。”
> “推荐为共享库设置版本号。”

在 kmcmake 中，这些不再需要文档。因为：
- GLOB 根本不存在。
- 链接作用域通过 `LINKS`（公开）和 `PLINKS`（私有）强制分离，没有模糊地带。
- 版本号通过 `configure_file` 自动处理，无需手动设置。

**管理意志通过工具落地，而非通过文档倡导。**

## 四、两类 `if`：规则系统的边界

kmcmake 的规则系统严格区分两类性质完全不同的条件判断。

### 类型 1：配置级 `if` —— 安全的参数选择

这类 `if` **不改变 target 的数量和依赖结构**，只影响编译参数、安装路径、目标属性。例如：

```cmake
if(KMCMAKE_ENABLE_SHARE)
    set(CMAKE_POSITION_INDEPENDENT_CODE ON)
endif()
```

这本质是**参数选择（parametric choice）**：构建图的结构不变，只是某些参数被赋予不同值。kmcmake **允许**这类 `if`，因为它们不会引入非确定性。

### 类型 2：结构级 `if` —— 危险的图变体

这类 `if` **改变 target 的存在与否、依赖关系、源文件列表**，导致构建图的结构随条件变化。例如：

```cmake
if(USE_CUDA)
    add_library(cuda_lib ...)
else()
    add_library(cpu_lib ...)
endif()
```

这是**结构变体（structural variant）**：构建图本身不再唯一。这类 `if` 是 kmcmake **严格禁止**的，因为它使构建系统从“声明式”堕落为“可编程脚本”，AI 无法静态推理，人类也难以维护。

**规则系统的边界就在这里：参数可以变，结构不能变。** 这一边界由工具强制，无需文档说明。

## 五、分层决策：构建系统定战略，源码做执行

kmcmake 规则系统的另一核心是**将决策权下沉到最合适的层级**。构建系统只负责“战略”层面的配置（如启用哪些特性），而“战术”层面的具体执行（某段代码是否编译）完全下沉到源码层，通过预处理宏精确控制。

这就像大公司的管理层级：**上层定义战略和方向，基层负责具体执行**。如果上层管理者每天盯着基层员工完成了多少行代码，那就是管理错位。同样，如果构建系统试图通过 `if` 分支精确控制每一个源文件的编译与否，那它就在做“微观管理”，导致构建脚本臃肿、难以维护、对 AI 极不友好。

kmcmake 的规则系统强制了这种分层：

- **构建系统层（上层）**：通过 `configure_file` 将配置选项写入 `version.h`，定义一组清晰的宏（如 `STATIC_LOAD_ICU`、`ENABLE_CUDA`）。它不关心这些宏具体在哪些 `.cc` 文件里被使用，只负责“颁布政策”。
- **源码层（基层）**：各个 `.cc` 文件根据 `version.h` 中的宏，用 `#ifdef` 精确控制自己是否参与编译、是否启用某段逻辑。这是局部、可控的决策，不会影响构建图的结构。

以 Goose 项目为例，其 `version.h` 中清晰定义了一组 `STATIC_LOAD_*` 宏：

```cpp
#define STATIC_LOAD_CORE_FUNCTION
#define STATIC_LOAD_JSON
#define STATIC_LOAD_PARQUET
// #define STATIC_LOAD_ICU   // 可选，由构建系统控制
```

源码中根据这些宏决定是否编译特定模块：

```cpp
#if defined(STATIC_LOAD_ICU)
        {
            "icu", [](Goose &db) {
                db.LoadStaticExtension<IcuExtension>();
                return ExtensionLoadResult::LOADED_EXTENSION;
            }
        },
#endif
```

这种设计带来多重好处：

1. **构建系统极度简洁**：没有任何结构级 `if` 控制 target 结构，构建图是**静态且唯一**的。
2. **配置完全透明**：开发者（或 AI）只需查看 `version.h`，就能一目了然地知道当前构建包含了哪些特性。
3. **规则不可绕过**：所有特性开关必须在 `version.h` 中定义，无法在构建脚本中临时“创造”一个条件分支。
4. **AI 友好**：AI 不需要推理构建脚本中的复杂分支，只需读取 `version.h` 的宏定义，就能理解整个项目的特性组合。

## 六、kmcmake 的本质：可执行的规则系统

kmcmake 不是一套“最佳实践”，而是一个**将管理意志编码为工具约束的规则系统**。它通过三层规则层层递进：

1. **语言层规则**：剥夺图灵完备性，从源头消除整类错误。
2. **宏层规则**：封闭表达空间，每个意图只有唯一表达。
3. **分层决策规则**：强制构建系统只做战略配置，源码做战术执行。

传统“设计哲学”是建议性的——你可以遵守，也可以不遵守。而 kmcmake 的规则是**强制性的**——你只能这样写，没有第二种写法。这就是**工具即制度**：制度不再是纸面上的文字，而是嵌入工具中的不可绕过约束。

| 传统方式 | kmcmake 规则系统 |
|---------|-----------------|
| 文档规定“不要用 GLOB” | GLOB 根本不存在 |
| 文档规定“明确链接作用域” | 通过 LINKS/PLINKS 强制分离 |
| 文档规定“版本号统一管理” | `configure_file` 自动处理 |
| 文档规定“特性开关写在 version.h” | 构建脚本无法绕过 version.h |
| 代码审查检查规范 | 工具层面无法违反 |

## 七、对 AI 的根本价值

kmcmake 将 CMake 的搜索空间从无限压缩到有限，从非确定性变为确定性。AI 的问题不是“不会写 CMake”，而是**图灵完备性导致的搜索空间爆炸**。

| 原生 CMake 的问题 | kmcmake 的解决方案 | 搜索空间压缩效果 |
|------------------|-------------------|-----------------|
| 同一个意图有几十种写法 | 每个意图只有一个宏 | 从 O(几十) 到 O(1) |
| 依赖管理分散、易遗漏 | 中心化 `KMCMAKE_DEPS_LINK` | 从 O(N) 到 O(1) |
| 编译器 flags 需要手动适配 | 自动根据编译器 ID 选择 | 从 O(几十) 到 O(1) |
| SIMD 指令集检测需要手写 | 内置检测，一个变量控制 | 从 O(几十) 到 O(1) |
| 安装和打包需要大量样板 | `PUBLIC` 参数 + 自动生成 | 从 O(几十行) 到 O(1) |
| 测试注册繁琐 | `kmcmake_cc_test` 一行搞定 | 从 O(几十行) 到 O(1) |
| 结构级控制流导致不确定性 | 完全禁止 if/foreach/动态生成 | 从“图灵完备”到“声明式” |

AI 不再需要理解 `target_link_libraries` 的 `PUBLIC/PRIVATE/INTERFACE` 语义，不再需要手写 `find_package` 和 `target_include_directories`，不再需要处理 `install` 和 `export`。所有复杂性被 kmcmake 宏封装，AI 的“犯错空间”被压缩到几乎为零。

## 九、工程实践：从零创建一个 kmcmake 项目

### 9.1 使用模板生成项目

```bash
cmake -S kmcmake/template -B build -DCHANGEME=myproject
cmake --install build --prefix /path/to/myproject
```

### 9.2 添加一个库

```cmake
set(SRCS src/a.cc src/b.cc)
kmcmake_cc_library(
    NAME myproject
    NAMESPACE ${PROJECT_NAME}
    SOURCES ${SRCS}
    LINKS ${KMCMAKE_DEPS_LINK}
    PUBLIC
)
```

### 9.3 添加一个测试

```cmake
kmcmake_cc_test(
    NAME mytest
    MODULE myproject
    SOURCES mytest.cc
    LINKS doctest::doctest myproject::myproject_static
)
```

AI 可以轻松生成上述代码，无需理解 CMake 的复杂语法。

## 十、与 xconfig、Goose 的统一

kmcmake、xconfig、Goose 共享同一套设计公理：**将管理意志编码为工具约束，通过规则系统消灭非确定性**。

| 项目 | 领域 | 规则系统体现 | 管理意志下沉 |
|------|------|-------------|-------------|
| xconfig | 配置管理 | 强类型系统、声明式校验器 | 类型安全、并发安全由框架强制 |
| kmcmake | 构建系统 | 剥夺图灵完备性、分层决策 | 构建规范由工具强制，无需文档 |
| Goose | SQL IR | 结构化 IR、显式优化规则 | SQL 语义由 IR 约束，优化路径唯一 |

**所有构建系统、配置管理、SQL 优化中的错误，有相当大比例来源于表达空间过大导致的非确定性选择问题。** kmcmake 的实践表明，**工具即制度**——将管理意志编码为可执行的规则系统，剥夺表达自由度，降级为刻意不完备的声明式子语言，并将决策权下沉到最合适的层级，是解决这一问题的根本路径。

## 十一、边界与演进：约束不是终点，而是方向

kmcmake 的核心主张是**通过约束消除非确定性**，但这并不意味着系统必须在一开始就达到“零控制流”的绝对形态。工程实践中，适度的灵活性是必要的，关键在于**明确约束的边界**。

当前 kmcmake 的边界可以概括为：

- **结构必须稳定**：构建图（target 数量、依赖关系）在配置阶段是唯一且确定的。这是**不可谈判的硬边界**。
- **参数可以变化**：允许通过配置级 `if` 或全局变量对编译参数、属性进行选择。这是**可协商的软边界**。
- **控制流逐步收敛**：不强制完全禁止 `if`，但限制其作用范围，避免侵入构建图定义。这是**演进的方向**。

这种设计背后的原则是：

> **优先约束“结构的不确定性”，再逐步收紧“参数的自由度”。**

换句话说，kmcmake 当前解决的是最大的问题——**构建图的非确定性**。而对于更细粒度的控制（如参数选择是否也应完全声明化），则作为后续演进方向，而非一开始的强制约束。

未来的演进路径可以包括：

- 将部分配置级 `if` 进一步收敛为声明式参数（减少控制流）
- 引入 lint / 静态分析，逐步限制 `if` 的使用范围
- 将更多“隐性参数选择”转化为显式配置（类似 `version.h` 模式）

最终目标不是“禁止一切”，而是：

> **让系统的表达逐步从“控制流驱动”收敛为“数据驱动”。**

这与全文贯穿的“工具即制度”一脉相承：制度不是一成不变的教条，而是随着工程实践不断收紧边界的演进过程。kmcmake 提供了一个**可生长的规则系统**——今天它约束了构建图，明天它可以约束更多。这或许才是约束系统最务实、也最可持续的形态。

## 结语

kmcmake 的实践表明，**工具即制度**——将管理意志编码为可执行的规则系统，剥夺表达自由度，降级为刻意不完备的声明式子语言，并将决策权下沉到最合适的层级，是解决构建系统非确定性问题的根本路径。希望这份记录能为 AI 时代的系统设计提供一份可参考的范式。
